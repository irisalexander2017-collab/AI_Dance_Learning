## 2026-07-03 — 正式编舞起点规则

- Date: 2026-07-03
- Request: 明确音乐分析与正式编舞起点的职责边界。
- Reason: 老师可能在正式编舞前加入 freestyle、准备动作或临时动作，系统无法仅凭音乐可靠判断正式编舞从哪里开始。
- Decision:
  - 系统只分析音乐拍点、BPM 和音乐八拍结构，不自动判断正式编舞起点。
  - 用户在视频播放过程中点击一次“正式编舞从这里开始”。
  - 系统将点击时间自动对齐到最近的音乐整数拍或半拍，并判断动作从第几拍或哪个 “and” 进入。
  - 系统从用户确认的位置划分学习段，并自动生成动作前四拍的预备计数。
  - 正式学习起点之前的视频继续保留，可通过“播放完整原视频”查看，但默认不进入学习八拍。
  - 音乐时间线保留真实拍点；学习段从用户选择的正式起点开始重新编号。
- Status: 已确认；手动标记、吸附和学习段已在 Phase 2A 实现。
- Planned phase: 后续自动节拍分析继续沿用该规则，并保留手动模式作为可靠后备。
- Classification: 后续产品需求，不是当前 Bug。

## 2026-07-03 — 竖屏放大拖动与全屏安全显示

- Date: 2026-07-03
- Request: 竖屏视频在“放大填充”时允许自由拖动；进入全屏时自动显示完整画面，退出后恢复原放大模式和位置。
- Decision:
  - 竖屏 cover 模式支持鼠标上下拖动，有横向溢出时也支持左右拖动。
  - 拖动位置按 cover 实际溢出空间限制，不能把画面拖离可视区域。
  - 增加“重置画面”，恢复默认居中但不改变播放状态。
  - 竖屏全屏临时强制使用 contain；用户保存的 cover 模式和拖动位置不被覆盖，退出全屏后自动恢复。
  - 横屏继续保持用户当前 contain/cover 模式，不套用竖屏全屏覆盖逻辑。
  - 第一版不再增加预设手部、脚部或上半身放大按钮。
- Status: 普通页面拖动、边界、重置和状态恢复已处理；全屏前后效果待用户手动验证。
- Classification: Phase 1.1 最后修正，不属于 Phase 2。

## 2026-07-03 — Phase 2A 手动校准八拍学习

- Date: 2026-07-03
- Request: 在不进行自动音乐分析的前提下，通过手动 BPM、音乐第一拍和用户确认的正式编舞起点建立八拍学习段。
- Decision:
  - 音乐时间线以真实秒数保存整数拍与半拍。
  - 正式编舞起点由用户标记并吸附到最近的整数拍或 `and`，系统不得自动判断。
  - 学习段从确认起点开始，每段持续完整 8 拍；结尾不足 8 拍时明确标记。
  - 数拍语音使用本地 Web Speech API，屏幕高亮与视频时间线才是准确依据。
  - 每个视频使用文件名、大小和最后修改时间组成独立存储键。
- Status: 已实现；循环播放听感、语音调度和全屏显示待用户本地验证。
- Classification: Phase 2A，不包含自动 BPM 或拍点识别。

## 2026-07-03 — Phase 2A.1 练习界面简化与全屏侧边栏

- Date: 2026-07-03
- Request: 保持视频持续可见，移除第一版数拍语音与拍点视觉层，并为全屏练习增加可折叠的侧边控制栏。
- Decision:
  - 删除 Web Speech API、整数/半拍/只预备/关闭模式、预备倒数和拍点高亮；这些不再属于当前运行逻辑。
  - 播放模式仅保留“循环练习”和“连续播放”。循环在范围结束后固定暂停 2 秒再重播；连续模式只在“当前段”范围自动前进下一段。
  - 普通页面的视频区域固定保持约 60vh 并随滚动保持可见；低频的节拍与分段设置默认展开，确认正式起点后自动收起。
  - 全屏使用包含视频、进度、状态和侧边栏的播放器根容器；侧边栏支持展开/收起和左右换边，位置保存在浏览器本地。
  - 竖屏全屏继续强制完整画面，退出后恢复普通页面的显示模式与拖动位置；横屏保持用户选择。
- Status: 已实现；真实媒体循环、连续播放和全屏交互需要用户在本地浏览器手动验证。
- Classification: Phase 2A.1 体验修正，不是 Phase 2B，也不包含自动音乐分析。

## 2026-07-03 — Phase 2A.1 Blocking Regression 修复

- Date: 2026-07-03
- Request: 修复 Phase 2A.1 重构引入的播放按钮缺失、竖屏旧裁切位置复用、显示模式/拖动/重置失效等阻挡性回归，同时保留八拍练习和全屏侧边栏。
- Root cause:
  - 普通页面播放按钮被错误放入“已有学习段”条件，尚未建立学习段时完全不渲染。
  - 竖屏拖动位置使用一个跨视频全局 localStorage 值，新视频会继承上一个视频的顶部/底部裁切坐标。
  - 全屏显示与普通页面共用同一显示/拖动状态，无法清晰表达“竖屏进入全屏先 contain、退出恢复普通页面状态”。
- Decision:
  - 普通页面恢复始终可见的播放/暂停控制，所有播放按钮继续连接同一个真实 video ref 和 `togglePlayback()`。
  - 拖动位置改为按视频存储；旧的全局坐标不再读取，新视频默认居中，同一视频重选可恢复并在 metadata 后重新限制。
  - 普通页面与全屏使用独立的临时显示模式和拖动位置；竖屏进入全屏先 contain，退出不会污染普通页面状态。
  - 保留固定 2 秒循环、连续下一段、学习范围、精简布局和全屏侧边栏。
- Status: 播放控制、竖屏显示状态、拖动边界和重置的数据流已修复；生产构建通过。普通页面真实媒体及全屏效果仍需用户本地验证。
- Classification: Phase 2A.1 引入的 Blocking Regression，不是新功能。
